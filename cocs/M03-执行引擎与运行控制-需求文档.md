## 1. 模块职责

- 负责：测试生命周期调度、套件遍历、步骤执行驱动、失败策略与结果汇总。
- 负责：管理 `TestOptions`、fixture 生命周期、用例级超时、退出状态。
- 不负责：YAML 解析细节、不负责具体动作实现、不负责绘制 Headed 面板。
- 输入/输出：输入为 `yamlPath`、`directory`、`TestOptions`、`ExecutionPlan`；输出为 `TestResult`、`TestSuiteResult`、运行期事件流。

## 2. 数据模型

### TestOptions（运行选项）

| 字段名 | 类型 | 必填/可选 | 语义 | 合法范围 | 默认值 |
| --- | --- | --- | --- | --- | --- |
| Headed | bool | 必填 | 是否启用可视化运行 | `true` 或 `false` | `true` |
| DefaultTimeoutMs | int | 必填 | 默认步骤超时 | 范围 `[100, 600000]`，包含端点 | `10000` |
| DebugOnFailure | bool | 必填 | 失败时是否保留现场 | `true` 或 `false` | `true` |
| ReportOutputPath | string | 必填 | 报告输出目录 | 非空字符串 | `"Reports"` |
| ScreenshotPath | string | 必填 | 截图输出目录 | 非空字符串 | `"Reports/Screenshots"` |
| StopOnFirstFailure | bool | 必填 | 套件失败是否立即停止 | `true` 或 `false` | `false` |
| ContinueOnStepFailure | bool | 必填 | 单用例步骤失败后是否继续 | `true` 或 `false` | `false` |
| ScreenshotOnFailure | bool | 必填 | 步骤失败时是否自动截图 | `true` 或 `false` | `true` |
| RetryCount | int | 可选 | 用例失败后的重试次数（V1 预留，不执行重试） | `null` 表示不重试；范围 `[0, 5]` | `null` |

## 3. CRUD 操作

| 操作 | 入口 | 禁用条件 | 实现标识 | Undo语义 |
| --- | --- | --- | --- | --- |
| 运行单个 YAML 用例 | `TestRunner.RunTest(string yamlPath, TestOptions options = null)` | `yamlPath` 为空；编译失败；测试上下文未初始化 | `TestRunner.RunTest` `(设计提案，实现时确认)` | 不涉及；只读执行 |
| 运行目录下全部 YAML | `TestRunner.RunSuite(string directory, TestOptions options = null)` | `directory` 不存在；无 `.yaml` 文件 | `TestRunner.RunSuite` `(设计提案，实现时确认)` | 不涉及；只读执行 |
| 执行单步骤 | 执行引擎遍历 `ExecutionPlan.steps` | 当前用例已终止；步骤超时；动作未注册 | `StepExecutor.ExecuteStep` `(设计提案，实现时确认)` | 不涉及；执行行为由具体动作自行处理 |
| 保留失败现场 | `DebugOnFailure=true` 且步骤失败 | `Headed=false` 且未启用失败快照；用户主动停止运行 | `FailureStateKeeper.Preserve` `(设计提案，实现时确认)` | 不涉及；只读保留 |
| 停止当前运行 | Headed 面板点击"停止"或命令行收到中止信号 | 当前无运行中的用例 | `RuntimeController.Stop` `(设计提案，实现时确认)` | 不涉及；停止后不执行剩余步骤 |

## 4. 交互规格

- 触发事件：测试入口调用 `RunTest` 或 `RunSuite`。
- 状态变化：`Pending -> Running -> Passed/Failed/Error/Skipped`。
- 数据提交时机：每个步骤执行完成后立即生成 `StepResult`；用例结束后一次性汇总 `TestResult`；套件结束后汇总 `TestSuiteResult`。
- 取消/回退：用户主动停止时，当前步骤允许完成本帧清理后中断；未开始的步骤一律不执行。
- 失败策略：`ContinueOnStepFailure=false` 时，任一步骤失败即停止当前用例并进入 `teardown`；`true` 时记录失败并继续后续步骤。
- 套件策略：`StopOnFirstFailure=true` 时，当前用例结束后停止后续用例遍历。
- 超时策略：单步骤超时后标记当前步骤为 `Failed`，错误码为 `STEP_TIMEOUT`；是否继续由 `ContinueOnStepFailure` 决定。
- 每帧驱动：执行引擎通过 `EditorApplication.update` 注册回调，在每帧回调中推进步骤执行，禁止 busy wait 或 `Thread.Sleep`（见 `TODO-004`）。

## 6. 校验规则

### 输入校验

| 规则 | 检查时机 | 级别 | 提示文案 |
| --- | --- | --- | --- |
| `DefaultTimeoutMs` 必须在合法范围内 | 创建 `TestOptions` 后、运行前 | Error | `DefaultTimeoutMs 超出允许范围` |
| `ReportOutputPath` 与 `ScreenshotPath` 必须为非空路径 | 运行前初始化输出目录时 | Error | `输出目录不能为空` |
| `RunSuite` 目录必须存在 | 遍历套件前 | Error | `测试目录不存在：{path}` |
| Headed 模式与批处理无头模式组合冲突时以显式参数为准 | 解析运行参数后 | Warning | `当前运行环境不支持 Headed，将自动关闭` |
| 当前用例必须有可执行步骤 | 进入步骤循环前 | Error | `测试用例 {name} 没有可执行步骤` |

### 错误响应

| 错误场景 | 错误码 | 错误消息模板 | 恢复行为 |
| --- | --- | --- | --- |
| 运行入口参数非法 | TEST_OPTIONS_INVALID | `测试运行参数非法：{detail}` | 抛异常并终止运行 |
| 测试目录不存在 | TEST_SUITE_DIRECTORY_NOT_FOUND | `测试目录不存在：{path}` | 抛异常并终止套件 |
| 当前步骤超时 | STEP_TIMEOUT | `步骤 {stepName} 执行超时：{timeoutMs}ms` | 记录失败；根据 `ContinueOnStepFailure` 决定是否继续 |
| 执行器抛未处理异常 | STEP_EXECUTION_EXCEPTION | `步骤 {stepName} 执行异常：{detail}` | 记录 `Error`，进入 `teardown` |
| 主动停止运行 | TEST_RUN_ABORTED | `测试运行已停止：{reason}` | 结束当前用例并标记未执行步骤为未开始 |
| 套件无 YAML 文件 | TEST_SUITE_EMPTY | `测试目录 {path} 中未找到 YAML 用例` | 返回 `TestSuiteResult.total=0`，不抛异常 |

## 8. 技术实现要点

- 关键类与职责：
  - `TestRunner` `(设计提案，实现时确认)`：单用例与套件入口；持有 `RuntimeController` 引用。
  - `StepExecutor` `(设计提案，实现时确认)`：调度动作执行、超时控制、结果封装。
  - `RuntimeController` `(设计提案，实现时确认)`：处理暂停、继续、停止状态；持有 `CancellationTokenSource`，停止时 Cancel。
  - `ExecutionContext` `(设计提案，实现时确认)`：在单次运行内共享根 `VisualElement`、数据行、选项和结果收集器。

- 核心流程：

```text
RunTest(yamlPath, options)
-> YamlTestCaseParser.ParseFile
-> ExecutionPlanBuilder.Build
-> Initialize ExecutionContext
-> Execute setup steps
-> Register EditorApplication.update callback
-> Per-frame: advance to next step, check timeout, check cancellation
-> Execute teardown steps
-> Aggregate TestResult
-> Emit report event
```

- 每帧驱动说明（更优方案）：在 EditMode 环境中，推荐使用 `EditorApplication.update` 注册每帧轮询回调替代协程，原因：
  1. EditMode 下 Unity 协程（`StartCoroutine`）依赖 `MonoBehaviour`，Editor 中不可直接使用。
  2. `EditorCoroutineUtility.StartCoroutineOwnerless` 可替代，但需要 `com.unity.editorcoroutines` 包额外安装。
  3. `EditorApplication.update` 是 Editor 内置支持，无需额外依赖，可直接用状态机推进步骤。
  4. `TODO-004` 需在 M0 PoC 中验证两种方案，最终选型在 PoC 结束后更新本文档。

- 性能约束：
  - 单步骤执行循环的轮询粒度固定为每帧一次，禁止 busy wait。
  - `StepResult` 写入必须增量完成，禁止用例结束后反推状态。
  - 套件遍历必须按文件名升序稳定执行，避免 CI 结果抖动。
  - 失败快照最多保留最近 `20` 个附件路径，避免单用例内存无限增长。

- 禁止项：
  - 禁止在执行器中直接解析 YAML 文本。
  - 禁止在 Headed 模式与无头模式之间复用同一个可变静态状态对象。
  - 禁止使用 `Thread.Sleep` 等阻塞 Unity 主线程的等待方式。

- TODO(待确认)：EditMode 驱动方案最终选型见 `TODO-004`，推荐优先验证 `EditorApplication.update` 状态机方案。

## 9. 验收标准

1. [存在合法 YAML 文件] -> [调用 `RunTest`] -> [返回 `TestResult.status=Passed`，且步骤结果数等于 YAML 步骤数]
2. [目录下存在 3 个 YAML 文件] -> [调用 `RunSuite`] -> [按文件名字典序执行 3 个用例并返回准确汇总计数]
3. [第 2 步执行失败，`ContinueOnStepFailure=false`] -> [执行用例] -> [第 3 步不执行，但 `teardown` 仍执行]
4. [第 2 步执行失败，`ContinueOnStepFailure=true`] -> [执行用例] -> [第 3 步继续执行，最终用例状态为 `Failed`]
5. [运行中用户点击停止] -> [执行当前用例] -> [当前运行在本帧清理后终止，后续未执行步骤不再运行]
6. [单步骤超时] -> [执行用例] -> [步骤结果错误码为 `STEP_TIMEOUT`，并根据 `ContinueOnStepFailure` 决定是否继续]
7. [套件目录存在但无 `.yaml` 文件] -> [调用 `RunSuite`] -> [返回 `TestSuiteResult.total=0`，不抛异常]

## 10. 边界规范

- 空数据：套件目录存在但没有 YAML 文件时，返回 `TestSuiteResult.total=0`，不抛未处理异常。
- 单元素：仅 1 个步骤的 `ExecutionPlan` 也必须完整产出 `startedAtUtc`、`endedAtUtc`、`durationMs`。
- 上下限临界值：`DefaultTimeoutMs=100` 与 `600000` 合法；超界立即以 `TEST_OPTIONS_INVALID` 阻断运行。
- 异常数据恢复：执行器内部抛出未处理异常时，必须仍尝试执行 `teardown`，并将用例状态记为 `Error`。

## 11. 周边可选功能

- P1：支持并行运行多个 YAML 套件；当前不预留共享状态，避免 V1 引入并发复杂度；扩展时需引入线程安全的 `TestSuiteResult` 聚合器。
- P1：支持失败重试；当前预留 `TestOptions.RetryCount` 字段（已在数据模型中声明），V1 收到非 `null` 值时报 `Warning` 并忽略，不进入重试逻辑。
- P2：支持步骤级性能埋点导出 JSON；当前预留 `StepResult.metrics: Dictionary<string, double>` 字段扩展点。
