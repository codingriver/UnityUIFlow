# UnityUIFlow 测试用例说明与编写规范

版本：1.3.0  
日期：2026-04-13  
状态：以当前测试代码为准更新覆盖口径

## 1. 文档目标

本文档用于说明 UnityUIFlow 当前测试体系的目录结构、覆盖范围、缺口边界，以及后续新增测试时应遵循的编写规则。

当前测试结论分为两层：

- 当前代码已覆盖：现有 MVP 实现、官方 `PanelSimulator` 动作链路、执行引擎、Headed 基础能力、报告与 CLI 主流程，以及复杂字段、菜单、命令、集合视图、Toolbar 等新增动作能力。
- 当前代码未完全覆盖：部分编辑器面板 UI，以及未直接继承 `EditorWindowUITestFixture<TWindow>` 的官方 fixture 基类路径。
- 当前代码已新增边界覆盖：官方 UI host / `PanelSimulator` 真实绑定、strict fail-fast、宿主/驱动报告标识、`RootOverrideOnly` 非官方入口边界、复杂字段 DSL 与动态子树控件自动化边界。

## 2. 当前测试目录结构

### 2.1 测试代码

目录：`Assets/Tests/`

当前文件职责如下：

- `UnityUIFlow.ParsingAndPlanningTests.cs`
  - 覆盖 YAML 解析、数据驱动、模板渲染、时长解析、选择器编译、执行计划构建、动作注册基础能力。
- `UnityUIFlow.LocatorsAndActionsTests.cs`
  - 覆盖元素定位、存在性与可见性语义差异、官方/兼容点击/双击/输入/按键/悬停/滚动/拖拽动作，以及 `set_value`、`select_option`、菜单动作、命令动作、集合视图动作、复杂字段动作、Toolbar 与动态子树控件动作。
- `UnityUIFlow.ExecutionReportingCliTests.cs`
  - 覆盖执行器主流程、失败策略、超时、报告输出、JSON 结果、CLI 参数解析、项目设置覆盖。
- `UnityUIFlow.HeadedTests.cs`
  - 覆盖 `RuntimeController`、`HeadedRunEventBus`、`HighlightOverlayRenderer` 基础链路，以及 Headed strict 开关入口与偏好持久化。
- `UnityUIFlow.ExamplesAcceptanceTests.cs`
  - 覆盖 `Assets/Examples` 下示例 YAML、自动宿主窗口、Examples 验收路径。
- `UnityUIFlow.TestHelpers.cs`
  - 提供 Unity 协程测试与 `Task` 之间的桥接工具。
- `UnityUIFlow.Tests.cs`
  - 占位文件，用于兼容工程与 asmdef 结构。

### 3.3 当前总判断

当前测试用例没有覆盖整个项目的全部功能。  
它们已经覆盖当前 MVP 主链路、官方 `PanelSimulator` 执行链、InputSystem 键盘桥接、真实截图链路、复杂字段 DSL、Toolbar/动态子树控件局部自动化，以及 `RootOverrideOnly`/strict fail-fast 边界，但还没有覆盖完整编辑器面板 UI，以及“直接继承 `EditorWindowUITestFixture<TWindow>`”这一条可选实现路径。

## 4. 当前未覆盖范围

以下区域当前仍属于明确缺口，新增功能开发时必须同步补测：

1. `EditorWindowUITestFixture<TWindow>`（来自 `com.unity.ui.test-framework@6.3.0`）的直接继承式生命周期、窗口创建与清理兼容性仍未单独补测。
2. `UnityUIFlowProjectSettingsProvider` 的界面交互与保存链路仍未覆盖。
4. `ToolbarPopupSearchField` 弹出结果列表、`ToolbarBreadcrumbs` 专用导航、`ObjectField` Object Picker / DragAndDrop、`CurveField` / `GradientField` 浮窗仍属于代码与测试都诚实保留的边界。

### 5.1 必增测试集

- 当前必须持续保留并扩展的官方 UI host / simulator 回归：
  - `click`
  - `double_click`
  - `hover`
  - `drag`
  - `scroll`

- InputSystem 高保真输入测试：
  - `press_key`
  - `type_text`
  - 焦点切换
  - 提交键 / 取消键

- 官方 fixture / 官方宿主测试：
  - `OfficialEditorWindowPanelSimulator` 宿主桥接
  - `EditorWindowUITestFixture<TWindow>` 直接继承路径
  - `SetUp` / `TearDown` 生命周期
  - 与 YAML 执行桥接
  - 与 Headed 共存

### 5.2 必须保留的迁移兼容测试

- 现有 fallback 动作测试不能立即删除。
- 迁移期必须同时保留：
  - 当前 fallback 行为回归
  - 官方驱动行为回归
- 当官方驱动全面接管后，fallback 测试转为兼容模式测试，不再作为正式交互语义验收主依据。

## 6. 新增测试用例编写规范

### 6.1 一条测试只验证一个核心结论

推荐示例：

- `OfficialClickAction_UsesUiTestFrameworkDriver`
- `TypeText_UsesInputSystemKeyboardPath`
- `Fixture_UsesOfficialEditorWindowHost`

不推荐把解析、执行、报告、截图、Headed、设置面板混在一个测试里同时验证。

### 6.2 优先断言结果，不只断言“没报错”

优先断言：

- 文本状态变化
- 控件值变化
- 焦点变化
- 滚动位置变化
- 拖拽结果变化
- 错误码
- 报告产物真实落盘

### 6.3 用例命名必须说明驱动语义

当测试验证的是官方链路还是 fallback 链路时，测试名必须写清楚：

- `..._Fallback...`
- `..._Official...`
- `..._InputSystem...`

### 6.4 YAML 回归与单元测试同时保留

- 单元/集成测试：
  - 快速定位底层逻辑问题。
- YAML 端到端回归：
  - 保证真实用户入口仍然可运行。

两类测试不能互相替代。

## 7. 测试维护建议

1. 每次新增一个动作能力，至少新增一条对应测试。
2. 每次修复一个 bug，优先先补失败复现测试，再改实现。
3. 涉及语义边界变化时，要同步更新：
   - 测试
   - `cocs/M05-动作系统与CSharp扩展-需求文档.md`
   - `cocs/M09-测试基座与Fixture基类-需求文档.md`
   - `cocs/M12-官方UI测试框架与输入系统测试接入-需求文档.md`
   - `cocs/TODO.md`
4. 迁移期不要把 fallback 通过误写成官方能力已完成。
5. 当前环境未探测到官方 UI host / simulator 时，必须保留 strict 模式失败测试，防止报告口径回退。

## 8. 推荐执行方式

本地至少执行以下验证：

- 编译验证
  - `dotnet build Assembly-CSharp.csproj`
  - `dotnet build Assembly-CSharp-Editor.csproj`
- Unity Test Runner 验证
  - EditMode 测试
  - 包含 `[UnityTest]` 的协程测试

接入官方框架后，建议新增一组单独的“官方驱动回归”测试分类，避免与 fallback 回归结果混淆。
