# UnityUIFlow 测试用例说明与编写规范

版本：1.1.0  
日期：2026-04-10  
状态：更新基线

## 1. 文档目标

本文档用于说明 UnityUIFlow 当前测试体系的目录结构、覆盖范围、缺口边界，以及后续新增测试时应遵循的编写规则。

当前测试结论分为两层：

- 当前代码已覆盖：现有 MVP 实现、fallback 动作链路、执行引擎、Headed 基础能力、报告与 CLI 主流程。
- 当前代码未覆盖：`com.unity.test-framework` UI 测试子系统正式接入、InputSystem 高保真输入链路、官方 fixture 宿主、真实截图方案、部分编辑器 UI。

## 2. 当前测试目录结构

### 2.1 测试代码

目录：`Assets/Tests/`

当前文件职责如下：

- `UnityUIFlow.ParsingAndPlanningTests.cs`
  - 覆盖 YAML 解析、数据驱动、模板渲染、时长解析、选择器编译、执行计划构建、动作注册基础能力。
- `UnityUIFlow.LocatorsAndActionsTests.cs`
  - 覆盖元素定位、存在性与可见性语义差异、fallback 点击/双击/输入/按键/悬停/滚动/拖拽动作。
- `UnityUIFlow.ExecutionReportingCliTests.cs`
  - 覆盖执行器主流程、失败策略、超时、报告输出、JSON 结果、CLI 参数解析、项目设置覆盖。
- `UnityUIFlow.HeadedTests.cs`
  - 覆盖 `RuntimeController`、`HeadedRunEventBus`、`HighlightOverlayRenderer` 基础链路。
- `UnityUIFlow.ExamplesAcceptanceTests.cs`
  - 覆盖 `Assets/Examples` 下示例 YAML、自动宿主窗口、Examples 验收路径。
- `UnityUIFlow.TestHelpers.cs`
  - 提供 Unity 协程测试与 `Task` 之间的桥接工具。
- `UnityUIFlow.Tests.cs`
  - 占位文件，用于兼容工程与 asmdef 结构。

### 2.2 测试资源

目录：

- `Assets/Examples/Editor/`
- `Assets/Examples/Uxml/`
- `Assets/Examples/Uss/`
- `Assets/Examples/Yaml/`

这些资源用于承载 `ExamplesAcceptanceTests` 的端到端回归。

## 3. 当前测试覆盖说明

### 3.1 按模块覆盖

| 模块/区域 | 当前覆盖结论 | 说明 |
| --- | --- | --- |

| Actions fallback 交互链路 | 已覆盖 | `click`、`double_click`、`type_text`、`type_text_fast`、`press_key`、`drag`、`hover`、`scroll` 等已回归 |
| Execution / RuntimeController | 已覆盖 | 执行器主流程、超时、失败策略、Headed 控制已覆盖 |

| `UnityUIFlowProjectSettingsProvider` 设置面板 UI | 未覆盖 | 当前仅覆盖设置应用逻辑，未覆盖 Provider 界面 |

### 3.3 当前总判断

当前测试用例没有覆盖整个项目的全部功能。  
它们已经覆盖了当前 MVP 主链路，但还没有覆盖这次新决策要求的“官方 UI 测试框架 + InputSystem 高保真输入 + 官方 fixture 宿主”三块核心能力。

## 4. 当前未覆盖范围

以下区域当前仍属于明确缺口，新增功能开发时必须同步补测：

1. `com.unity.test-framework@1.7.0` UI 测试子系统的初始化、官方交互驱动绑定与运行回归。
2. 基于 `com.unity.inputsystem` 的测试输入设备创建、键盘输入、文本输入、快捷键与焦点链路。
3. `EditorWindowUITestFixture<TWindow>` 的真实生命周期、窗口创建与清理兼容性。
4. `ScreenshotManager` 的真实窗口截图，而不是当前 2x2 占位 PNG。
5. `BatchRunnerWindow` 的编辑器面板交互、批量运行状态同步与路径偏好持久化。
6. `UnityUIFlowProjectSettingsProvider` 的界面交互与保存链路。
7. 官方驱动与 fallback 兼容模式的切换测试。

## 5. 接入官方框架后的新增测试要求

### 5.1 必增测试集

- 官方 UI 驱动动作测试（`com.unity.test-framework` UI 测试子系统）：
  - `click`
  - `double_click`
  - `hover`
  - `drag`
  - `scroll`

- InputSystem 高保真输入测试：
  - `press_key`
  - `type_text`
  - 焦点切换
  - 提交键 / 取消键

- 官方 fixture 宿主测试：
  - `EditorWindowUITestFixture<TWindow>` 创建
  - `SetUp` / `TearDown` 生命周期
  - 与 YAML 执行桥接
  - 与 Headed 共存

### 5.2 必须保留的迁移兼容测试

- 现有 fallback 动作测试不能立即删除。
- 迁移期必须同时保留：
  - 当前 fallback 行为回归
  - 官方驱动行为回归
- 当官方驱动全面接管后，fallback 测试转为兼容模式测试，不再作为正式交互语义验收主依据。

## 6. 新增测试用例编写规范

### 6.1 一条测试只验证一个核心结论

推荐示例：

- `OfficialClickAction_UsesUiTestFrameworkDriver`
- `TypeText_UsesInputSystemKeyboardPath`
- `Fixture_UsesOfficialEditorWindowHost`

不推荐把解析、执行、报告、截图、Headed、设置面板混在一个测试里同时验证。

### 6.2 优先断言结果，不只断言“没报错”

优先断言：

- 文本状态变化
- 控件值变化
- 焦点变化
- 滚动位置变化
- 拖拽结果变化
- 错误码
- 报告产物真实落盘

### 6.3 用例命名必须说明驱动语义

当测试验证的是官方链路还是 fallback 链路时，测试名必须写清楚：

- `..._Fallback...`
- `..._Official...`
- `..._InputSystem...`

### 6.4 YAML 回归与单元测试同时保留

- 单元/集成测试：
  - 快速定位底层逻辑问题。
- YAML 端到端回归：
  - 保证真实用户入口仍然可运行。

两类测试不能互相替代。

## 7. 测试维护建议

1. 每次新增一个动作能力，至少新增一条对应测试。
2. 每次修复一个 bug，优先先补失败复现测试，再改实现。
3. 涉及语义边界变化时，要同步更新：
   - 测试
   - `cocs/M05-动作系统与CSharp扩展-需求文档.md`
   - `cocs/M09-测试基座与Fixture基类-需求文档.md`
   - `cocs/M12-官方UI测试框架与输入系统测试接入-需求文档.md`
   - `cocs/TODO.md`
4. 迁移期不要把 fallback 通过误写成官方能力已完成。

## 8. 推荐执行方式

本地至少执行以下验证：

- 编译验证
  - `dotnet build Assembly-CSharp.csproj`
  - `dotnet build Assembly-CSharp-Editor.csproj`
- Unity Test Runner 验证
  - EditMode 测试
  - 包含 `[UnityTest]` 的协程测试

接入官方框架后，建议新增一组单独的“官方驱动回归”测试分类，避免与 fallback 回归结果混淆。
